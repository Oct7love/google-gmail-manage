## 一句话

一个**只读**多账号 Gmail 查看器 Mac 桌面应用，服务一位 Noon 电商卖家管理约 30 个会频繁轮换的 Gmail 账号。

## 用户画像（非常重要）

- **非开发者**，中文母语
- Noon 电商卖家，有多仓库业务
- 用户的 PRD 习惯：文档要先充分提问、用他的语言描述需求，**不要从代码角度自说自话**
- 会用浏览器、会用 Chrome 扩展，但不会修代码
- 默认**用中文**和他沟通，除非他切换语言

## 核心业务约束（不要自作主张扩大范围）

用户已**明确拒绝**的功能，不要加：

| 已拒绝 | 为什么记录在这 |
|---|---|
| 发送邮件 | 用户明确说"不需要" |
| 跨账号搜索 | 用户明确说"只需要查看该账号接收到的邮件" |
| 按发件人/主题过滤 | 同上 |
| 附件查看 | 同上 |
| 已读/未读标记 | 同上 |
| 长历史邮件（>10 封） | 用户说"拉最近 10 封"，账号用几天就删 |
| 系统弹窗通知 | 用户选了 App 内显示未读数就够 |
| 跨账号统一收件箱 | 同"跨账号搜索"，用户只按账号独立看 |

任何"顺手加一下 XX"的念头，先打住。这个产品的价值就是**极简**，用户明确要极简。

## 技术选型（定了，别轻易改）

- **Electron + React + TypeScript** — 打包 `.app` 分发；Claude 最熟；迭代快
- **better-sqlite3** — 本地缓存；同步 API 不用写 async
- **googleapis** 官方 Node SDK — 最不容易被标记为异常客户端
- **macOS Keychain**（通过 `keytar` 或等价封装）— OAuth token 存这里，不要落盘明文
- **OAuth 2.0 Loopback 流**（Desktop client）— 不是 Web 流、不是 Device 流

### 为什么不是 Tauri / 原生 SwiftUI
Tauri 的 Rust 端 Claude 改起来慢；SwiftUI 迭代效率低、要 Xcode。Electron 是本项目下 Claude 协作最顺的选择。用户已接受这个权衡。

## 反封控原则（用户最在意的）

用户原话："不要轻易触发封控，一定要是阅读谷歌自己开放的 api"。

**做到：**
- 只用 `gmail.readonly` 一个 scope，别的都不要
- 调 API 走官方 `googleapis` 包，别自己拼 HTTP 请求
- User-Agent 用合理值；不要伪装浏览器
- 默认 **1 小时**后台刷新；不要加更高频的"心跳"
- 指数退避处理 429/5xx
- 授权失效不要自动"悄悄重试"——要在 UI 提示用户手动重新登录

**不要做：**
- 爬网页版 Gmail（mail.google.com）
- 模拟浏览器自动化
- 在后台持续探测账号状态
- 拉全部邮件历史

## OAuth 的 7 天坑（核心设计约束）

Google 对未通过审核的敏感权限应用实行**每 7 天 refresh token 过期**。用户**已知情并接受**，因为他的账号本来也每几天轮换。

所以：
- 每个账号在 UI 上要能**清楚显示**"已授权 / 授权已过期，请重新登录"
- 过期后的重新授权路径要**一键触发**（点那个账号 → 弹 OAuth 窗）
- 后台刷新遇到过期要静默记录，**不要反复重试消耗配额**

## 如果你要开始写代码

1. 先完整读一遍 `docs/implementation_plan.md`，按阶段推进
2. 用户不会自己跑命令，凡是需要他操作的（如 Google Cloud Console 配置、生成 OAuth client 凭据），**必须写清步骤 + 截图位置**
3. 每完成一个阶段，跑一次 `npm run build`，把 `.app` 能不能双击打开给用户看
4. 绝不静默加范围——想加任何 PRD 没有的功能，先问用户
