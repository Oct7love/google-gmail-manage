# 产品需求文档（PRD）

> 这份文档是本项目的"宪法"。所有技术决定、界面设计、功能取舍都要回到这里来核对。
> 如果哪个功能在本文档里找不到根据，那它就不应该出现在产品里。

---

## 1. 背景

用户是一位 Noon 电商卖家，出于业务原因（多店铺、多仓库、不同供应链对接）长期持有**约 30 个 Gmail 账号**。这些账号的生命周期不长——**每隔几天就会有一批新加入、一批被淘汰**。

目前他查看这些邮件的方式是在 Chrome 浏览器里一个一个切换登录，这带来三个问题：

1. **效率低**：每次切换要退出上一个账号、登入下一个，一轮下来几十分钟
2. **容易冲突**：多个 Google 账号同时在浏览器里切换，容易被 Google 判定为异常行为
3. **没有全景**：无法一眼看到哪些账号有新邮件

他需要一个**能在一个 Mac 应用里集中查看所有账号收件箱的工具**。

## 2. 用户画像

| 维度 | 描述 |
|---|---|
| 职业 | Noon 电商卖家，中文母语 |
| 技术背景 | 非开发者，会用常规软件、浏览器扩展 |
| 操作系统 | macOS |
| 邮箱数量 | 约 30 个 Gmail，每几天轮换 |
| 核心诉求 | **只读**多账号 Gmail，减少反复登录 |
| 风险担忧 | 不要触发 Google 的账号封控 |

## 3. 产品定位

**"一个极简的多账号 Gmail 只读查看器"**。

关键词：
- **极简** — 不是要做另一个完整邮件客户端
- **多账号** — 核心价值在于账号数量
- **只读** — 只接收不发送，权限最小化，最安全
- **查看器** — 不做搜索、过滤、整理、归档这类邮件管理功能

## 4. 目标（Goals）

- **G1**：用户能在一个 Mac 窗口里看到所有已绑定 Gmail 账号的最新邮件
- **G2**：添加一个新 Gmail 账号的操作步骤不超过 5 步、不超过 1 分钟
- **G3**：删除账号一键完成，且本地数据与 Google 端授权一并清除
- **G4**：后台自动定期同步新邮件，无需手动操作
- **G5**：整个数据通路只走 Google 官方 Gmail API，不触碰任何非官方渠道
- **G6**：30 个账号同时运行，整体 API 调用量远低于 Google 免费配额

## 5. 非目标（Non-Goals，明确不做）

以下功能**当前版本不做**，即使实现起来不难：

| 不做的功能 | 理由 |
|---|---|
| 发送邮件 | 用户明确不需要；更高权限增加封控风险 |
| 跨 30 个账号统一搜索 | 用户明确说"只需要查看该账号接收到的邮件" |
| 按发件人、主题、日期筛选 | 同上 |
| 附件下载/预览 | 同上 |
| 已读/未读切换、打星、归档、删除 | 只读产品，不改 Gmail 状态 |
| 标签、分组、规则 | 同上 |
| 邮件本地全文搜索 | 同上 |
| 系统弹窗通知（macOS Notification） | 用户选择只在 App 内显示未读数量 |
| 超过最近 10 封的邮件历史 | 账号几天就换，拉多了是浪费 |
| Windows / Linux 版本 | 用户只用 Mac |

> 这些功能不是"V2 再说"的留白，而是**设计上的主动选择**。要加之前必须回到 PRD 讨论。

## 6. 功能需求（Functional Requirements）

### FR-1：账号管理

- **FR-1.1** 首次打开 App，界面显示"未添加账号"的空状态，明显的"添加 Gmail 账号"按钮
- **FR-1.2** 点击"添加账号"→ 系统默认浏览器弹出 Google OAuth 登录页 → 用户输入邮箱、密码、2FA（如有）→ 同意 `gmail.readonly` 权限 → 浏览器跳回本地 loopback → App 接住 token → 账号出现在左侧列表
- **FR-1.3** 左侧账号列表显示邮箱地址（截断）+ 未读数字徽章
- **FR-1.4** 右键（或悬停出现"…"菜单）每个账号支持：**重新授权**、**移除账号**
- **FR-1.5** 移除账号时：弹二次确认 → 调用 Google OAuth revoke 接口撤销 token → 清除本地该账号所有缓存邮件 → 从 Keychain 删除凭据 → 从 UI 消失
- **FR-1.6** 若某账号的 token 已过期（7 天规则），该账号在列表里显示"授权已过期"图标，点击直接触发重新授权流程

### FR-2：邮件查看

- **FR-2.1** 点击左侧某账号 → 中间栏显示该账号**最近 10 封邮件**的标题列表（发件人、主题、时间）
- **FR-2.2** 点击某封邮件 → 右栏显示邮件正文（HTML 或纯文本，按邮件实际格式渲染）
- **FR-2.3** 正文里的外部链接点击时提示用户确认是否在浏览器打开（防钓鱼点击）
- **FR-2.4** 邮件列表默认按时间倒序
- **FR-2.5** 支持"加载更多"按钮，每点一次再拉 10 封（按需加载，不自动拉多）

### FR-3：刷新

- **FR-3.1** 后台每 **1 小时** 自动拉一轮所有账号的新邮件
- **FR-3.2** 当前选中账号的顶部有"手动刷新"按钮，点一次立刻拉
- **FR-3.3** "全部刷新"按钮在左侧账号列表顶部，点一次触发所有账号同步
- **FR-3.4** 刷新进行中显示旋转图标，完成后图标消失

### FR-4：初次配置（Google Cloud 凭据）

- **FR-4.1** 第一次启动 App 或未配置凭据时，显示"Google Cloud 凭据未配置"的引导页
- **FR-4.2** 引导页包含**分步指引**（图文结合）：去 Google Cloud Console 创建项目 → 启用 Gmail API → 配置 OAuth consent screen（External / Testing）→ 创建 OAuth Client ID（Desktop app 类型）→ 把 Client ID 和 Client Secret 粘贴到 App 输入框
- **FR-4.3** 粘贴后 App 验证凭据格式正确，保存到 Keychain

## 7. 非功能需求（Non-Functional Requirements）

### NFR-1：安全

- OAuth 的 `refresh_token` 和 `access_token` 只存 macOS Keychain，**不**写入本地文件/SQLite 明文字段
- Google Cloud OAuth Client Secret 也存 Keychain
- 邮件正文本地缓存到 SQLite，文件存在用户 Library 目录（系统级权限保护）
- 删除账号时 **调用** `https://oauth2.googleapis.com/revoke` 显式撤销 token，不只是本地删除

### NFR-2：反封控

- **唯一数据通路**：Google 官方 `googleapis` Node SDK 调 Gmail API
- **不使用**任何网页抓取、浏览器自动化、非公开 API
- Scope **只申请** `https://www.googleapis.com/auth/gmail.readonly`
- API 调用遵循 Google 推荐实践：指数退避重试、尊重 429 的 Retry-After 头
- 不发出超出正常节奏的请求（1 小时一轮，30 账号并发上限 5）
- 不在后台主动"探测"账号是否有效，只在用户操作或定时任务时调用

### NFR-3：性能

- App 冷启动 ≤ 3 秒（到看到界面）
- 切换账号时，中间栏显示缓存邮件 ≤ 300ms
- 所有 30 个账号的一轮刷新 ≤ 60 秒（按 5 并发估）
- 打包后 `.app` 体积 ≤ 200 MB

### NFR-4：可维护性

- 代码 100% TypeScript
- 主进程（Node/Electron main）与渲染进程（React）职责分明
- 所有 Google API 调用收敛在 `backend/gmail/*` 一个模块里，方便将来替换
- 所有文档用中文维护，与代码同仓库

## 8. 关键约束（Constraints）

### C-1：OAuth 7 天过期

Google 对"未经审核"的敏感权限应用实行**refresh token 每 7 天过期**的限制。该项目**不会**申请 Google 审核（审核需要 15000 美元/年 CASA 安全评估，非企业级项目不可行）。

**对用户意味着：每个账号每 7 天要重新授权一次**。

用户已知情并接受。产品设计需围绕此约束：
- UI 明显提示哪些账号已过期
- 重新授权成本一键化
- 不把"token 永不过期"作为隐含假设

### C-2：Google Cloud Console 一次性配置

用户必须自己在 Google Cloud Console 创建 OAuth 凭据。这是 Google 的规则，任何第三方工具都绕不过去。产品通过**图文引导**降低首次配置门槛。

### C-3：Testing 模式用户上限

OAuth Consent Screen 的 Testing 状态下，最多可添加 100 个"test user"（就是你用来授权的 Gmail 账号本身）。30 个账号远小于 100，不构成限制。

## 9. 成功标准（Success Criteria）

用户发布后第一个月，若以下都成立即为成功：

- [ ] 用户连续 4 周使用该 App 查看邮件，未因 App bug 退回浏览器方式
- [ ] 添加新账号的成功率 ≥ 95%（不含 Google 自身拒绝）
- [ ] 30 个账号的单日 API 调用总量 < Google 免费配额的 1%
- [ ] 无账号被 Google 判定为异常或封控
- [ ] 用户每周重新授权操作的总耗时 ≤ 20 分钟

## 10. 风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|---|---|---|---|
| Google 政策变化，敏感 scope 规则收紧 | 中 | 大 | 架构上把 API 调用封装，能快速切换到其他方案（如 IMAP + app password） |
| 某账号莫名被风控 | 低 | 中 | 只用官方 API + 低频调用是最优的预防；真出事也和本 App 无关 |
| Client Secret 泄露 | 低 | 中 | 存 Keychain；用户自己的项目，泄露只影响自己 |
| Electron 安全漏洞 | 低 | 中 | 禁用 nodeIntegration、开启 contextIsolation、所有 renderer 走 IPC |

## 11. 版本范围（v1.0）

本 PRD 描述的所有功能属于 **v1.0**。没有 v0.x 的精简版——MVP 就是这套最小集。

完成 v1.0 后，是否继续迭代由用户决定。**没有隐含的 v2 roadmap**。
