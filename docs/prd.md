# 产品需求文档（PRD）

> 这份文档是本项目的"宪法"。所有技术决定、界面设计、功能取舍都要回到这里来核对。
> 如果哪个功能在本文档里找不到根据，那它就不应该出现在产品里。

---

## 1. 背景

用户是一位 Noon 电商卖家，出于业务原因（多店铺、多仓库、不同供应链对接）长期持有**约 30 个 Gmail 账号**。这些账号的生命周期不长——**每隔几天就会有一批新加入、一批被淘汰**。

目前他查看这些邮件的方式是在 Chrome 浏览器里一个一个切换登录，这带来三个问题：

1. **效率低**：每次切换要退出上一个账号、登入下一个，一轮下来几十分钟
2. **容易冲突**：多个 Google 账号同时在浏览器里切换，容易被 Google 判定为异常行为
3. **没有全景**：无法一眼看到哪些账号有新邮件

他需要一个**能在一个桌面应用里集中查看所有账号收件箱的工具**。

## 2. 用户画像

| 维度 | 描述 |
|---|---|
| 职业 | Noon 电商卖家，中文母语 |
| 技术背景 | 非开发者，会用常规软件 |
| 操作系统 | macOS 自用，朋友可能用 Windows |
| 邮箱数量 | 约 30 个 Gmail，每几天轮换 |
| 核心诉求 | **只读**多账号 Gmail，减少反复登录 |
| 风险担忧 | 不要触发 Google 的账号封控 |

## 3. 产品定位

**"一个极简的多账号 Gmail 只读查看器，附带一些让日常更顺手的小工具"**。

关键词：
- **极简** — 不做另一个完整邮件客户端
- **多账号** — 核心价值
- **只读** — 只接收，不发送，权限最小化
- **查看器** — 不做搜索、过滤、整理、归档这类管理功能

## 4. 目标（Goals）

- **G1**：用户能在一个窗口里看到所有已绑定 Gmail 账号的最新 20 封邮件
- **G2**：添加一个新 Gmail 账号的操作步骤不超过 5 步（不含首次生成应用密码）
- **G3**：删除账号一键完成，清除本地所有数据
- **G4**：**实时推送**新邮件（IMAP IDLE，延迟 ~30 秒），同时保留 1 小时兜底轮询
- **G5**：整个数据通路走 IMAP 官方协议（`imap.gmail.com:993`），和主流邮件客户端行为一致
- **G6**：30 个账号同时运行，日常使用不会触发 Google 风控
- **G7**：电脑睡眠唤醒后自动恢复实时推送，不丢邮件

## 5. 非目标（Non-Goals，明确不做）

| 不做的功能 | 理由 |
|---|---|
| 发送邮件 | 用户明确不需要；权限越少越安全 |
| 跨 30 个账号统一搜索 | 用户明确说"只需要查看该账号接收到的邮件" |
| 按发件人、主题、日期筛选 | 同上 |
| 附件下载/预览 | 同上 |
| 已读/未读切换、打星、归档、删除 | 只读产品，不改 Gmail 状态 |
| 标签、分组、规则 | 同上 |
| 邮件本地全文搜索 | 同上 |
| 系统弹窗通知 | 用户选择只在 App 内显示未读数 |
| 超过 20 封邮件历史 | 账号几天就换，拉多了是浪费 |
| 存 Google 登录密码 | 安全红线。只存应用专用密码（权限仅限 IMAP） |
| 在 App 内用 Google 账号登录第三方站 | 用户讨论过，撞 Google 反欺诈，放弃 |

> 这些功能不是"V2 再说"的留白，而是**设计上的主动选择**。要加之前必须回到 PRD 讨论。

## 6. 功能需求（Functional Requirements）

### FR-1：账号管理

- **FR-1.1** 首次打开 App，显示空状态引导图标 + "添加账号"按钮
- **FR-1.2** 点击"添加账号"→ 弹出对话框：
  - 左侧：表单（Gmail 地址、应用专用密码、可选 TOTP 密钥）
  - 右侧：嵌入 Google 应用密码生成页 webview
- **FR-1.3** 用户在右侧 webview 中登录对应 Gmail → 生成 16 位应用密码 → 复制到左侧表单
- **FR-1.4** 点"验证并添加"→ App 用 IMAP 尝试登录 → 成功则入库，失败显示错误
- **FR-1.5** 支持**批量导入**：粘贴一行 `账号 密码 辅邮 2fa` 或 `账号----密码----辅邮----2fa----链接`，App 自动解析 → Gmail 和 2FA 密钥填入对应输入框，Google 登录密码和辅邮显示在"登录辅助"区（供复制到 webview）
- **FR-1.6** 成功后：表单和 TOTP 面板自动清空，webview 登出 Google，焦点回到邮箱输入框（方便继续添加下一个）
- **FR-1.7** 左侧账号列表显示头像（首字母 + 确定性颜色）+ 状态点 + 邮箱地址
- **FR-1.8** 每个账号的 "⋯" 菜单：**更新应用密码**、**移除账号**
- **FR-1.9** 移除账号时：弹二次确认 → 清 Keychain → 清本地缓存邮件 → 从 UI 消失
- **FR-1.10** 应用密码失效时（IMAP 返回 AUTHENTICATIONFAILED）：账号图标变 ⚠️，点击直接弹出"更新应用密码"对话框

### FR-2：邮件查看

- **FR-2.1** 点击左侧账号 → 中栏显示该账号**最近 20 封邮件**（标题、发件人、日期、摘要）
- **FR-2.2** 点击某封邮件 → 右栏显示正文
- **FR-2.3** 正文在 **sandboxed iframe** 里渲染：CSP 禁 JS、禁外部资源，允许 inline 样式
- **FR-2.4** 外部图片默认**不加载**（防追踪像素），顶部有"加载外部图片"按钮切换
- **FR-2.5** **内嵌图片（cid:）默认加载**（附件形式，不是追踪）
- **FR-2.6** 一键**翻译为中文**按钮，调用 Google Translate 免费端点，结果显示在正文上方的折叠卡片里

### FR-3：刷新

- **FR-3.1** **实时推送**：每账号一个持久 IMAP IDLE 连接，Gmail 推送新邮件后 ~30 秒内到达
- **FR-3.2** 新邮件到达时左栏账号右侧弹出橙色 "+N" 胶囊徽章（脉动 5 秒后消失，点开账号也会清掉）
- **FR-3.3** 后台每 **1 小时**兜底同步一次所有账号，防 IDLE 漏推
- **FR-3.4** 左栏顶部 🔄 = 立即同步全部账号（并发上限 5）
- **FR-3.5** 中栏顶部 🔄 = 仅同步当前账号
- **FR-3.6** 刷新中的账号头像显示旋转图标
- **FR-3.7** 电脑睡眠唤醒 → 所有 IDLE 自动重连 + 追赶同步

### FR-4：辅助工具

- **FR-4.1** **2FA 码生成器**（TotpPanel）：粘贴 base32 密钥或 otpauth:// URI → 本地生成 6 位码，30 秒倒计时，一键复制
- **FR-4.2** TotpPanel 为**受控组件**，用户可直接覆盖密钥并实时看到新码。修改后显示"已改动"徽章 + "还原"按钮
- **FR-4.3** 添加账号时保存的 2FA 密钥存到 Keychain（service `MailViewer-account-info`）
- **FR-4.4** **凭据抽屉**（CredentialsDrawer）：中栏 🔑 按钮打开，显示该账号的邮箱 / 应用密码 / Google 密码 / 2FA 密钥 + 实时码 / 辅邮 / 链接。默认打码，单个或整体显示。支持编辑保存
- **FR-4.5** **2FA 设置快捷跳转**：添加对话框 webview 工具栏的"→ 2FA 设置"按钮直达 `myaccount.google.com/signinoptions/two-step-verification`
- **FR-4.6** **粘贴一行解析**：支持 `账号 密码 辅邮 2fa` 和 `账号----密码----辅邮----2fa----链接` 两种格式，解析后自动填邮箱 + 2FA 密钥，登录密码/辅邮/链接显示在"登录辅助"区供复制

### FR-5：网络与兼容

- **FR-5.1** **webview 代理**：添加对话框右侧齿轮按钮可配置专用代理（HTTP 或 SOCKS5），持久化在 `userData/settings.json`，仅影响 webview 不影响 IMAP
- **FR-5.2** **连接失败兜底**：webview `did-fail-load` 时覆盖错误卡，提供"重试 / 配代理 / 系统浏览器"三个出口

## 7. 非功能需求（Non-Functional Requirements）

### NFR-1：安全

- **应用专用密码**只存 Keychain（macOS）/ Credential Manager（Windows），不写明文文件
- 邮件正文缓存在 SQLite（userData 目录，系统级权限保护）
- 删除账号时删 Keychain 条目（用户仍可去 Google 账号设置手动撤销应用密码）
- 邮件 HTML 强制走 sandboxed iframe + CSP，禁 JS，禁表单

### NFR-2：反封控

- 唯一数据通路：`imap.gmail.com:993` IMAP 协议 + 应用专用密码
- 客户端行为和 Mac Mail / Thunderbird / Outlook **无法区分**
- 调用节奏：1 小时一轮，30 账号并发上限 5
- 不做任何"探测"类请求
- 遇到 AUTHENTICATIONFAILED：停止重试，等用户更新密码

### NFR-3：性能

- App 冷启动 ≤ 3 秒
- 切换账号显示缓存邮件 ≤ 300ms
- 单个账号同步 20 封邮件 ≤ 5 秒（一次 IMAP 会话）
- 30 账号全部同步 ≤ 60 秒
- 打包体积 ≤ 250 MB

### NFR-4：可维护性

- 代码 100% TypeScript 严格模式
- 主进程（Node/Electron main）与渲染进程（React）通过 preload 白名单 IPC 通信
- 所有 IMAP 调用收敛在 `src/main/imap/` 一个目录
- 所有文档中文维护，与代码同仓库

### NFR-5：跨平台

- Mac：主力平台，macOS 13+
- Windows：用户朋友用，Windows 10+
- Linux：不做

## 8. 关键约束（Constraints）

### C-1：需要 Gmail 账号开启 2FA

应用专用密码的前提是账号开了 2FA。用户已知情：
- 如果某账号还没开 2FA，加的时候就加不进去
- 界面里 UI 会说明

### C-2：Google 风控账号救不回来

部分批量来源的 Gmail 账号已被 Google 风控，生成不了应用密码。这种账号**无论什么方案都救不回来**，不是本 App 能解决的。

### C-3：翻译依赖 Google 网络

国内访问 `translate.googleapis.com` 可能需要代理。翻译失败不影响邮件读取。

### C-4：IDLE 推送延迟由 Gmail 决定

Gmail 服务器批量推送 IDLE 通知，实测 5-60 秒。这是 Gmail 行为，非客户端可控。对比 Gmail API Push（可达到秒级），但该方案需要 Google Cloud Console + OAuth，受 test user 和 7 天过期双重限制，已在架构决策中放弃。

## 9. 成功标准（Success Criteria）

- [ ] 用户日常用本 App 替代浏览器切号
- [ ] 30 个账号一个月内无异常封控（排除账号本身问题）
- [ ] 添加新账号成功率 ≥ 95%（账号能开 2FA 的前提下）
- [ ] 朋友（Windows 用户）能下载 `.exe` 装上直接使用

## 10. 风险与应对

| 风险 | 应对 |
|---|---|
| Google 改 IMAP 政策 | 极少见。IMAP 是国际标准，影响面大 |
| 批量账号被风控 | 无解，账号本身问题 |
| 翻译接口改动 | 加错误兜底；必要时换其他端点 |
| Electron 安全漏洞 | 定期升级；已开 contextIsolation + sandbox iframe |

## 11. 版本范围

当前实现：**核心功能 + 辅助工具 + 打包分发** 全部就绪。

**产物**：
- Mac `.dmg`（arm64，~100MB）
- Mac `.app`（unpacked，~263MB）
- Windows `.exe`（x64，可交叉编译）
- 自定义应用图标（蓝色渐变信封 + 橙色通知点，与 UI 顶栏 Logo 同款）

**资源占用（生产模式）**：
- 总内存 ~412 MB（4 个 Electron 进程）
- 已经接近 Electron 下限；更低只能重写为 Tauri
