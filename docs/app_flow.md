# 应用流程（App Flow）

> 从用户视角描述"打开 App 会发生什么"，每个按钮点下去的流程长什么样。
> 本文档**不**讲代码实现，只讲用户看到和感受到的东西。实现细节看 `backend_structure.md`。

---

## 总览图

```
  [首次启动]──未配置凭据──►【引导：配置 Google Cloud 凭据】
      │                              │ 粘贴凭据
      └──已配置凭据 + 无账号──►【空状态：添加账号按钮】
                                     │ 点添加账号
                                     ▼
                           【OAuth 浏览器流程】
                                     │ 授权成功
                                     ▼
  ┌──────────────────【主界面：三栏】───────────────────┐
  │  左栏（账号列表）│ 中栏（邮件列表） │ 右栏（邮件正文） │
  └─────────────────────────────────────────────────────┘
        ▲
        │ 后台每 1 小时自动刷新；用户可点账号 / 刷新 / 添加 / 删除
```

---

## Flow 1：首次启动（未配置 Google Cloud 凭据）

**触发**：用户第一次双击打开 `.app`，本地 Keychain 里没有 Client ID / Client Secret。

**界面路径**：

1. App 窗口打开，显示**引导页**，大标题："一次性设置 Google Cloud 凭据"
2. 引导页包含 6 步分步说明（每步配一张参考截图）：
   1. 访问 Google Cloud Console（点按钮自动打开浏览器）
   2. 新建一个 Project（例如命名 "MailViewer"）
   3. 启用 Gmail API
   4. 配置 OAuth consent screen：选 External → 填 App 名字、用户邮箱 → 添加 scope `gmail.readonly` → 添加 test users（就是要管的那 30 个邮箱）
   5. 创建 OAuth Client ID：类型选 **Desktop app**
   6. 下载 JSON 或复制 Client ID + Client Secret
3. 引导页底部有两个输入框：**Client ID**、**Client Secret**，外加"导入 JSON 文件"按钮
4. 用户粘贴或导入 → 点"保存并开始"
5. App 验证格式（Client ID 以 `.apps.googleusercontent.com` 结尾、Secret 非空）
6. 凭据写入 Keychain → 进入 Flow 2

**错误路径**：
- 格式不对 → 输入框下方红字提示"Client ID 格式有误"，不切页
- Keychain 写入失败（极少见）→ 弹对话框提示用户手动授权钥匙串访问

## Flow 2：空状态（已有凭据，无账号）

**界面**：主窗口三栏布局，但：
- 左栏中央显示一个大加号按钮 + 文字 "添加你的第一个 Gmail 账号"
- 中栏、右栏灰色占位："选择左边的账号查看邮件"

**交互**：
- 点加号按钮 → 触发 Flow 3

---

## Flow 3：添加账号（OAuth 流程）

**触发**：用户点"添加账号"按钮。

**流程**：

1. App 启动本地 loopback HTTP server（端口由系统分配，例如 `http://127.0.0.1:53821`）
2. App 构造 Google OAuth 授权 URL：
   - `client_id` 取自 Keychain
   - `redirect_uri` 指向刚启动的 loopback server
   - `scope` = `https://www.googleapis.com/auth/gmail.readonly`
   - `access_type=offline`、`prompt=consent`（确保每次都拿到 refresh_token）
3. App 调系统默认浏览器打开这个 URL（**不用内嵌 webview**，避免 Google 封禁）
4. 浏览器显示 Google 登录页 → 用户选 / 输入账号、密码、2FA
5. 显示同意页："MailViewer 想访问你的 Gmail（只读）"→ 用户点"允许"
6. 浏览器被重定向到 `http://127.0.0.1:53821/?code=xxx&state=xxx`
7. App 的 loopback server 接住请求，取出 `code`
8. App 用 `code` 调 Google token 端点换取 `access_token` + `refresh_token`
9. App 调 Gmail API `users.getProfile` 拿到真实 email 地址
10. App 把 `refresh_token` 和 `access_token` 存 Keychain（key 为 email），email 存 SQLite 的账号表
11. App 立即拉该账号最近 10 封邮件缓存到 SQLite
12. 账号出现在左栏列表 → 自动选中 → 中栏显示邮件列表

**错误路径**：

| 情况 | 处理 |
|---|---|
| 用户在浏览器点"取消" | loopback 收到 `error=access_denied` → App 静默取消，不报错 |
| 用户关浏览器窗口不操作 | loopback 超时 60 秒 → 弹提示"未收到授权，请重试" |
| 同一个 email 再次添加 | 检测到已存在 → 弹对话框"已添加，是否重新授权？" |
| Google 返回 token 换取失败 | 弹对话框显示 Google 返回的错误信息 |
| 网络错误 | 弹对话框 + 重试按钮 |

## Flow 4：查看邮件（主流程）

**前提**：至少有一个账号已添加。

**界面**：三栏布局。

### 左栏：账号列表

每个账号一行：

```
[🟢] a****@gmail.com      [3]    ←  绿点 = 正常，[3] = 未读数
[🟢] b****@gmail.com
[⚠️] c****@gmail.com      [重新授权]   ← 7 天过期
[🟢] d****@gmail.com      [7]
...
```

顶部有两个按钮：**+ 添加账号**、**🔄 全部刷新**。

### 中栏：邮件列表

点某个账号后：

```
━━━ a****@gmail.com ━━━  [🔄 刷新]
──────────────────────────
 Noon Marketplace            14:32
 订单 #10234 已发货
──────────────────────────
 Supplier Co.               10:15
 报价单 Q2
──────────────────────────
 Google                      昨天
 安全警报：新设备登录
──────────────────────────
   ...（共 10 封）

  [加载更多]
```

### 右栏：邮件正文

点某封邮件后：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
订单 #10234 已发货
From: Noon Marketplace <noreply@noon.com>
To: a****@gmail.com
2026-04-22 14:32
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

你好，你的订单已发货……
（HTML 或纯文本正文）
```

**关键交互规则**：
- 点击正文里的链接 → **不直接跳**，先弹小对话框："即将在浏览器中打开：https://... 是否继续？"（防钓鱼）
- 正文里的外部图片**默认不加载**（防止跟踪像素），顶部有"加载图片"按钮
- 正文区可上下滚动、可选中复制，但不可编辑

## Flow 5：手动刷新

**入口 1：单账号刷新**
- 选中某账号，中栏顶部点 🔄
- 仅刷新该账号
- 图标变旋转 → 完成后恢复

**入口 2：全部刷新**
- 左栏顶部点 🔄
- 依次刷新所有账号（并发上限 5）
- 左栏每行账号在自己那一行显示独立旋转图标

---

## Flow 6：后台自动刷新

**规则**：App 启动后，每 1 小时触发一次，对所有已绑定账号执行刷新。

**流程**：
1. 定时器触发
2. 按 5 并发分批调 Gmail API
3. 每个账号拉最新邮件列表 → 对比本地缓存 → 只拉新的
4. 遇到过期账号 → 跳过 + 在 UI 标记该账号"授权已过期"
5. 遇到 429 / 5xx → 指数退避重试最多 3 次
6. 全部完成后更新 UI 未读数

**用户感受**：不打扰。没有弹窗、没有声音、没有 Dock 跳动。只是下次点过来时，邮件列表已经是新的。

---

## Flow 7：处理"授权已过期"

**触发**：后台或手动刷新时某账号返回 `invalid_grant`（refresh token 已过期）。

**UI 表现**：
- 该账号在左栏的图标从 🟢 变成 ⚠️
- 悬停显示 tooltip "授权已过期，点击重新登录"
- 点击该账号 → 不切到邮件视图，而是弹出"该账号需要重新授权"对话框
- 用户点"重新授权"→ 走 Flow 3（OAuth 流程），但 redirect 后不是新增账号，而是更新已有账号的 token

**实现要点**：
- 后台遇到过期**不要循环重试**（会浪费 API 配额）
- 标记后等用户主动操作

---

## Flow 8：删除账号

**触发**：用户在左栏账号行点 "…" 菜单 → "移除账号"。

**流程**：
1. 弹二次确认对话框："移除后本地邮件缓存将删除，授权也会在 Google 端被撤销。确定？"
2. 用户点"确定" →
3. App 调 `https://oauth2.googleapis.com/revoke?token=<refresh_token>` 撤销 token
4. 从 Keychain 删除该账号的 token
5. 从 SQLite 删除该账号的所有缓存邮件
6. 从 SQLite 的账号表删除该条记录
7. 该账号从左栏消失
8. 如果被删的是当前选中账号，中栏、右栏回到占位状态

**错误路径**：
- revoke 调用失败（网络问题）→ 依然清理本地，在 App 日志记录一条"revoke 失败"，提示用户手动去 Google 账号设置移除
- 用户在第 1 步点"取消" → 不做任何事

---

## Flow 9：App 关闭 / 重启

**关闭**：
- 点窗口红叉 → App 退出（Mac 默认 Dock 图标也消失，不做托盘常驻）
- 若刷新进行中 → 等 5 秒完成，超时强退
- 未保存的状态（当前选中账号）写入本地 settings.json 下次启动恢复

**重启**：
- 读 Keychain 拿所有账号的 token
- 读 SQLite 拿账号列表 + 缓存邮件
- 直接渲染左栏 + 中栏（上次选中的账号）
- 后台异步触发一次刷新

---

## Flow 10：凭据更换 / 重置

**入口**：菜单栏 → "设置" → "Google Cloud 凭据" → "更换"

**流程**：
- 弹对话框："更换凭据会让所有已添加账号失效（你需要重新授权每个账号）。确定？"
- 用户确认 → 清空 Keychain 里所有账号 token + 清空邮件缓存 → 回到 Flow 1 的引导页
- 账号列表仍保留（email 记录），但都标记为"需要重新授权"

---

## 边界行为约定

| 情况 | 行为 |
|---|---|
| 网络离线 | 不报错、显示缓存；顶栏小图标"离线中"；网络恢复自动同步 |
| Gmail API 429 | 指数退避，当前账号标"限流中"，2 分钟后重试 |
| 邮件正文超过 5 MB | 只渲染前 1 MB，底部提示"内容已截断" |
| 收件箱为空 | 中栏显示"暂无邮件" |
| SQLite 文件损坏 | 启动时检测到 → 弹"数据缓存已重置"，重建空数据库，凭据和 token 不动 |
